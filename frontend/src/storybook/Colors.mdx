<Title>Color Mapping</Title>
<Subtitle>Role-based semantic tokens for the modern track, with light/dark behaviour</Subtitle>

> **This is a living document.** Every new feature, component, or design change that introduces a new color usage, adds a token, or shifts an existing token's meaning must update this page in the same change. The same applies to `docs/modern-track-color-system.md` in the repo root. Treat updates to both surfaces as part of the definition-of-done for any color or theming change. Stale color docs cause real engineering cost: contributors reach for the wrong token, reviewers don't know what's canonical, and the system drifts.

The modern track uses **role-based semantic tokens** as the preferred way to apply color. Each token resolves to a different raw `cinemata-*` palette step in light vs dark theme, so components don't need `dark:` variants for theme awareness. Components reach for the raw palette only for genuine one-offs (carousel indicator dots, etc.).

The modern profile page is a representative composed surface: its banner,
header, multi-page tab navigation, cards, loading states, and focus rings use
the existing `bg/*`, `text/*`, `border/*`, and `ring/focus` roles. It introduces
one profile-specific token, `text-text-profile-section-icon`, for the section
emblem glyphs (see Text tokens below); everything else routes through shared
roles rather than new aliases.

This page is the working reference for the token catalogue. For the underlying design rationale, the body-scoped wiring pattern, and the full architectural overview (palette, legacy aliases, modern tokens), see `docs/modern-track-color-system.md` in the repo root.

<Heading>When to reach past the tokens</Heading>

A small number of patterns are genuinely unique and stay on the raw `cinemata-*` palette. These are the documented exceptions:

- **Switch track and thumb** — `bg-switch-track-on` (light: `coral-reef-600`, dark: `coral-reef-700`), `bg-switch-track-off` (light: `pacific-deep-900`, dark: `pacific-deep-800`), `bg-switch-thumb` (`neutral-100`). Component-unique semantic tokens for the autoplay switch recipe.
- **Avatar inverted-neutral inner bg** — `bg-cinemata-neutral-200 dark:bg-cinemata-neutral-50`. INVERTS neutrals between themes for the initials-fallback ring.
- **Carousel indicator dots** — `bg-cinemata-strait-blue-800 dark:bg-white` (active), `bg-cinemata-pacific-deep-400/30 dark:bg-white/30` (inactive). One-off "active indicator on media surface" pattern.
- **HeroSection retry button** — `bg-cinemata-strait-blue-700` solid with `-800` hover. Bespoke decorative button.
- **Carousel overlay arrow text** — theme-aware blue pair `strait-blue-700 → strait-blue-600p`. Large chrome controls overlaid on media; no other shared use.

For all other usages, prefer a semantic token. If you find yourself reaching for the same raw palette pair more than once for the same role, consider whether it deserves a new token.

<Heading>Auth page feature tokens (account / mfa)</Heading>

The redesigned auth pages (`templates/account/`, `templates/mfa/`) render in the legacy Django shell but consume the modern token layer. They use the semantic tokens above directly wherever a token already carries the needed value — `bg-bg-page`, `bg-bg-surface` (gradient start), `text-text-strong`, `text-text-muted`, `text-text-danger`, `ring-ring-focus`, `text-text-link`, and `var(--bg-primary)` for the primary button + header icon. **Fields are transparent** (no surface fill) so the gradient panel shows through, separated only by the underline border.

Auth **components** use purpose-based semantic tokens (uniform with the rest of the app), not `--auth-*` feature aliases: accent/CTA button → `bg-secondary`/`bg-secondary-hover`; primary & secondary buttons → `bg-primary`/`bg-primary-hover`; button text → `text-on-primary`; checkbox checked fill → `bg-secondary` (same as the shared `CheckboxButton`); field underline → `border-default`; requirements panel → `bg-surface-muted` / `text-secondary`; status badges → `bg-success-subtle` / `bg-danger-subtle` (new, see Background tokens) + `text-success` / `text-danger`.

Only one genuinely auth-specific **decorative** feature token remains on `.auth-figma-page` (the panel gradient end stop now uses `bg-page` directly):

- **`--auth-ring-color`** — decorative concentric ring outlines. Light: `pacific-deep-200` (coincides with `border-default`) / Dark: `pacific-deep-500` (no existing token has this dark, kept lighter than `border-default`'s `pacific-deep-700` so the faint rings stay visible).

The full-screen message overlay defines a parallel set on `.auth-message-stack` (scrim, banner gradient, `color-mix` ring/shadow, success/error/info icon badges) — see `tailwind.css`.

> The `TextField` shared component exposes a **`transparent`** prop that mirrors this pattern in React: it clears the shell `bg-surface` so the input inherits the surface behind it.

> Aliases that resolve to an existing token in both themes were removed in favour of the existing token: `--auth-page-bg` → `bg-page`, `--auth-field-bg` → `bg-surface`, `--auth-panel-from` → `bg-surface`, `--auth-text-strong` → `text-strong`, `--auth-icon-color` → `bg-primary`, `--auth-button-primary-bg/-hover` → `bg-primary/-hover`, `--auth-text-muted` → `text-muted` (this last one differed only in light but coincided after `text-muted` moved to `pacific-deep-500`), `--auth-panel-to` → `bg-page` (dark already matched; light follows).

<Heading>Legacy semantic aliases (still valid, narrower scope)</Heading>

The pre-existing legacy semantic aliases inherited from MediaCMS (`bg-surface-popup`, `text-content-body`, `bg-brand-primary`, etc.) are still wired and still work. They map to the legacy `--btn-*`, `--body-*`, `--site-*` variables in the SCSS theme files. New code in `features/` should generally prefer the role-based tokens above, but the legacy aliases remain useful as bridges into legacy components.

Notable bridge: **`bg-brand-primary`** is intentionally redirected to `var(--bg-secondary)` (orange) so the modern utility and the legacy `--btn-primary-bg-color` SCSS variable can diverge. The modern `.bg-brand-primary` utility renders orange (light: `sunset-horizon-400p`, dark: `sunset-horizon-500`) while legacy server-rendered buttons styled by SCSS rules referencing `--btn-primary-bg-color` keep their historical blue (light) / orange (dark) values.

Video player theme: the `--site-player-*` family (`-accent-color`, `-progress-color`, `-track-color`, `-loaded-color`, `-tooltip-bg-color`, `-control-color`, `-canvas-color`, `-control-surface-color`, `-subtitle-bg-color`, `-subtitle-color`, `-shadow-color`, exposed as `bg-site-player-*` / `text-site-player-*` utilities) is the shared, theme-aware accent and control-surface system for the media view page player and the home hero player. Progress / volume / subtitle accent uses sunset-horizon (500 light / 400p dark), the big play button uses strait-blue (500 light / 300 dark), the buffered range uses pacific-deep, the seek tooltip uses pacific-deep-950, control and subtitle text use white, and the video canvas uses black. The hero player consumes these tokens directly instead of declaring its own color aliases, and the legacy `VideoPlayer.scss` consumes them as plain `var(--site-player-*)` so the player no longer follows the configurable site `--brand-color`.

<Heading>Which palette scale plays which role</Heading>

- **pacific-deep** — themed surfaces and headings (text 700–900 light, surfaces 800–950 dark). Most theme-aware tokens build on this scale.
- **strait-blue** — primary action color in the modern semantic layer (`bg/primary`), heading/body text on dark surfaces (`strait-blue-50`), nav active states.
- **sunset-horizon** — secondary action color in the semantic layer (`bg/secondary`, what `bg-brand-primary` routes through), inline links (`text/link`), focus rings (`ring/focus`). The brand's orange.
- **coral-reef** — focused input border in light mode, progress bars.
- **neutral** — themed surfaces and text in light mode (50–200 backgrounds, 500–900 text); dark-mode subtle text fallback.
- **red / green / amber** — semantic feedback (errors / success / warnings).
- **white / black** — text on colored backgrounds, dialog overlays, badge text.

<Heading>Source files</Heading>

- `frontend/src/static/css/config/_light_theme.scss` — All `--cinemata-*` palette values on `body` (light mode), plus light-mode overrides for the modern semantic intermediate vars.
- `frontend/src/static/css/config/_dark_theme.scss` — Dark-mode overrides on `body.dark_theme` for both the palette singletons and the modern semantic vars.
- `frontend/src/static/css/tailwind.css` — Defines the modern semantic intermediate vars on `body` and `body.dark_theme`, then exposes them via `@theme inline` as `--color-bg-*`, `--color-text-*`, `--color-border-*`, `--color-ring-*`. Also bridges the legacy `--color-brand-primary` through `var(--bg-secondary)`.
- `docs/modern-track-color-system.md` — Architectural overview of all three layers (palette, legacy aliases, modern tokens), body-scoped wiring pattern, and rationale.
